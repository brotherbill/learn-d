# 00.10.20 — Coding Discipline  
## micro-steps.md  
### The Micro‑Steps of Correctness‑First Development

Correctness‑first developers do not “jump around” in code.  
They move in **micro‑steps** — tiny, controlled, deterministic actions that prevent monsters from being born.

This file defines the discipline.

---

## 1. One Micro‑Step at a Time  
A micro‑step is:

- one change  
- one idea  
- one invariant  
- one truth  
- one movement of the cursor  
- one movement of the buffer  
- one movement of the state machine  

Never two.

Two steps at once create ambiguity.  
Ambiguity creates monsters.

---

## 2. Micro‑Steps Prevent Undefined Behavior  
Undefined behavior is a spawning ground.  
Micro‑steps eliminate it by ensuring:

- every change is understood  
- every change is verified  
- every change is reversible  
- every change preserves invariants  

Undefined behavior cannot hide when the developer moves slowly and deliberately.

---

## 3. Micro‑Steps Prevent Multiple Sources of Truth  
Truth fractures when:

- two changes happen at once  
- two ideas are mixed  
- two invariants are modified together  
- two states are updated without isolation  

Micro‑steps enforce **singular truth**.

One step → one truth.

---

## 4. Micro‑Steps Make Debugging Trivial  
Debugging is hard when:

- many changes happened at once  
- the developer cannot isolate the cause  
- the system moved in leaps instead of steps  

Micro‑steps make debugging trivial:

- the last step is always the cause  
- the last step is always visible  
- the last step is always reversible  

Debugging becomes a mechanical process, not a guessing game.

---

## 5. Micro‑Steps Make Code Review Safe  
Reviewers can understand:

- one change  
- one invariant  
- one truth  

Reviewers cannot understand:

- ten changes  
- ten invariants  
- ten truths  

Micro‑steps produce code that is readable, reviewable, and correct.

---

## 6. Micro‑Steps Prevent Monster Factories  
Monster factories include:

- caching authoritative truth  
- try/catch hiding failures  
- race conditions  
- eventual consistency  
- dynamic memory surprises  
- async/await state machines  
- multiple databases  
- undefined behavior  

Micro‑steps prevent these by forcing the developer to:

- see every detail  
- understand every mechanism  
- avoid every snake pit  
- preserve every invariant  

Correctness‑first development is micro‑step development.

---

## 7. Micro‑Steps Are the Opposite of Whiz‑Bang  
Whiz‑bang features encourage:

- big leaps  
- hidden behavior  
- implicit state  
- implicit concurrency  
- implicit allocation  
- implicit failure modes  

Micro‑steps require:

- explicit behavior  
- explicit state  
- explicit memory  
- explicit failure paths  
- explicit invariants  

If NASA won’t use a feature, correctness‑first developers ask why.  
The answer is usually: **it hides too much to allow micro‑steps.**

---

## 8. Micro‑Steps Are the Foundation of Professional Engineering  
Professionals:

- move slowly  
- move deliberately  
- move predictably  
- move deterministically  

Amateurs:

- jump  
- guess  
- hope  
- patch  

Micro‑steps are the difference.

---

## 9. Summary  
Micro‑steps enforce correctness by:

- preventing undefined behavior  
- preventing truth from fracturing  
- preventing monster spawning grounds  
- preventing irreproducible errors  
- preventing hidden state transitions  
- preventing snake pits  
- preventing leaps into the unknown  

Correctness‑first developers move in micro‑steps.  
This is how professionals build systems that matter.
