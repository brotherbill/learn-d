Coding discipline is the set of behaviors, habits, and constraints that prevent errors from being born in the first place.  
It is the practical expression of correctness‑first thinking.

Correctness‑first explains **why** discipline is necessary.  
Coding discipline explains **how** professionals achieve it.

This module teaches the discipline required to build systems that remain correct as they grow.

---

## 1. Discipline Is the Difference Between Professionals and Amateurs  
### Amateurs write code. Professionals build systems.

Anyone can write code that “works once.”  
Professionals write code that:

- works every time  
- works under load  
- works in production  
- works after future changes  
- works when integrated with other systems  
- works without spawning little monsters (multiple sources of truth)  

Coding discipline is the difference between:

- *accidental success* and *predictable correctness*  
- *debugging chaos* and *stable foundations*  
- *patch‑driven development* and *contract‑driven development*  

Discipline is not optional.  
It is the cost of correctness.

---

## 2. What Coding Discipline Actually Means  
### It is not style. It is not preference. It is not aesthetics.

Coding discipline is:

- enforcing contracts  
- maintaining invariants  
- building on correct foundations  
- eliminating undefined behavior  
- preventing truth from fracturing  
- refusing to allow multiple sources of truth  
- writing code that cannot lie  

It is the daily practice of correctness‑first.

---

## 3. Knowing Exactly How Things Work  
### Professionals understand the machinery beneath their tools.

Correctness‑first developers do not rely on magic.  
They understand the mechanics underneath every operation.

In **01‑write‑writeln/** we demonstrate:

- how buffers work  
- how cursors advance  
- how output is staged  
- why `write("something");` does **not** immediately display on the screen  

No student who studies that module will ever be surprised when:

```d
write("something");
debugger_breakpoint();
```

…shows nothing on the terminal.

They know **exactly** why:

- the buffer has not flushed  
- the cursor has not advanced  
- the terminal has not been updated  

This is coding discipline:  
**understanding the mechanics so nothing is mysterious.**

Professionals do not guess.  
They know.

## 4. Using Simple, Well‑Understood Tools  
### Correctness‑first developers avoid snake pits disguised as modern conveniences.

Modern languages offer features that look elegant but hide complexity:

- `=>` lambda sugar  
- `async/await` concurrency abstractions  
- dynamic arrays with hidden reallocations  
- garbage‑collected memory with unpredictable pauses  

These tools often conceal:

- dynamic memory allocation  
- hidden state machines  
- implicit concurrency  
- unpredictable performance  
- undefined behavior under load  

Correctness‑first developers prefer **simple tools** that are:

- well understood  
- predictable  
- mechanically reasoned  
- free of hidden traps  
- safe under all conditions  

#### 4a. Simple Tools Are Accessible and Fast

Simple tools:

- can be understood by junior programmers  
- behave predictably  
- run fast  
- avoid hidden allocations  
- avoid hidden concurrency  
- avoid hidden state transitions  

They make correctness easier to achieve.

#### 4b. Classic Tools Without Dynamic Memory Are Precise and Reliable

Classic tools that avoid dynamic memory and garbage collection:

- are precise  
- are fast  
- are reliable  
- do exactly what they say  
- never surprise the developer  
- never hide a snake pit under window dressing  

They “just work,” and they work **safely**.

---

## 4c. Discipline Means Not Jumping to Every Whiz‑Bang Feature  
### Understand it fully — or leave it alone.

Coding discipline is not chasing every new (or old) whiz‑bang feature that appears in a language.  
It is not adopting tools because they look clever, modern, or fashionable.

A correctness‑first developer follows a strict rule:

> **Do not use a feature until you understand it fully.  
Only then can you judge whether it brings advantages without a snake pit.**

Many language features offer small conveniences while dragging along enormous hidden complexity:

- implicit state machines  
- hidden concurrency  
- dynamic memory allocation  
- garbage‑collection pauses  
- unpredictable performance  
- undefined behavior under load  

If a feature brings:

- a tiny convenience  
- a small syntactic sugar  
- a minor reduction in keystrokes  

…but also brings a **snake pit** of hidden behavior, then discipline says:

> **Leave it alone. Snakes aren’t cozy and cuddly.**

Correctness‑first developers choose tools that:

- are predictable  
- are mechanically reasoned  
- have no hidden traps  
- behave the same under all conditions  
- make correctness easier, not harder  

This is not conservatism.  
This is engineering.

The goal is not to avoid modern tools.  
The goal is to avoid **snake pits disguised as modern tools**.

A disciplined developer evaluates every feature with one question:

> **Does this make correctness easier, or harder?**

If the answer is “harder,” the feature is rejected — no matter how fashionable it is.

This is coding discipline.

## 5. Discipline Prevents the Birth of Monsters  
### Incorrect systems create monsters. Discipline prevents them.

Incorrect systems allow:

- mismatched records  
- inconsistent states  
- contradictory ledgers  
- duplicated truth  
- corrupted data  
- fractured invariants  

These monsters are expensive, dangerous, and contagious.

Coding discipline prevents them from being born.

A disciplined system:

- has one source of truth  
- maintains its invariants  
- enforces its contracts  
- rejects invalid states  
- refuses to produce contradictions  

Discipline is the shield that protects the system from exponential error growth.

---

## 6. Discipline Is Cheaper Than Chaos  
### The economics are not subtle.

Undisciplined code leads to:

- regressions  
- outages  
- reconciliation departments  
- audits  
- customer compensation  
- compliance failures  
- emergency patches  
- organizational panic  

Disciplined code leads to:

- predictable behavior  
- stable foundations  
- linear growth  
- reliable features  
- trustworthy systems  
- calm operations  

The cost difference is enormous.

Discipline is cheaper than chaos.

---

## 7. Discipline Scales, Debugging Does Not  
### Human cognition is linear. Errors are exponential.

Debugging does not scale.  
Discipline does.

A disciplined system:

- grows linearly  
- remains predictable  
- remains correct  
- remains stable  

An undisciplined system:

- grows exponentially in error  
- becomes unpredictable  
- becomes fragile  
- becomes unfixable  

Coding discipline is the only scalable engineering strategy.
