# 00.10.30 — Major Steps  
## The Major‑Step Discipline of Professional Engineering

Micro‑steps prevent monsters from being born.  
Major‑steps prevent *systems* from drifting into chaos.

A **major‑step** is a controlled, correctness‑first transition from one stable state to another.  
It is the opposite of “big bang” changes, rewrites, leaps of faith, or whiz‑bang migrations.

Major‑steps are how professionals move a system forward without spawning little monsters.

---

## 1. A Major‑Step Is a Transition Between Two Stable States  
A major‑step is:

- one stable invariant →  
- one deliberate transition →  
- one new stable invariant

Never two transitions at once.

Two transitions at once create:

- undefined behavior  
- fractured truth  
- irreproducible errors  
- monster factories  

Correctness‑first developers move from stability to stability.

---

## 2. Major‑Steps Require Full Mechanical Understanding  
A major‑step cannot be attempted until the developer understands:

- the current state machine  
- the future state machine  
- the invariants of both  
- the failure modes of both  
- the memory and allocation behavior  
- the concurrency behavior  
- the error‑handling behavior  

If any part is unclear, the major‑step is forbidden.

Correctness‑first developers do not migrate into the unknown.

---

## 3. Major‑Steps Are Built from Micro‑Steps  
A major‑step is not a leap.  
It is a **sequence** of micro‑steps:

- each micro‑step preserves invariants  
- each micro‑step is reversible  
- each micro‑step is mechanically understood  
- each micro‑step is verified  

A major‑step is simply the *aggregation* of many micro‑steps, each correct, each safe.

Professionals do not jump.  
They advance.

---

## 4. Major‑Steps Must Not Introduce New Sources of Truth  
A major‑step must never:

- add a second database  
- add a second ledger  
- add a second cache  
- add a second “eventually consistent” replica  
- add a second authority for truth  

Major‑steps must **reduce** sources of truth, not increase them.

Every additional truth is a spawning ground for little monsters.

---

## 5. Major‑Steps Must Not Introduce Snake Pits  
A major‑step must not introduce:

- async/await state machines  
- dynamic memory surprises  
- garbage‑collection pauses  
- try/catch exception paths  
- hidden concurrency  
- hidden allocation  
- hidden recomputation  
- hidden caching  

If NASA won’t use it, correctness‑first developers ask why.  
The answer is usually: **it hides too much to be safe in a major‑step.**

---

## 6. Major‑Steps Must Preserve All Invariants  
A major‑step must preserve:

- ledger invariants  
- buffer invariants  
- cursor invariants  
- state‑machine invariants  
- memory invariants  
- concurrency invariants  
- correctness invariants  

If an invariant must be broken temporarily at key points, the major‑step is invalid.  
Correctness‑first developers do not “break invariants at key points and fix them later.”

Later is where monsters live.

---

## 7. Major‑Steps Must Not Produce Irreproducible Errors  
In large organizations, only **about 30% of error tickets are reproducible**.  
The remaining 70%:

- cannot be reproduced  
- cannot be diagnosed  
- cannot be traced  
- cannot be explained  
- cannot be fixed  

This is what happens when major‑steps are done incorrectly.

Irreproducible errors mean:

- the system continued after something impossible happened  
- undefined behavior was allowed  
- invariants were violated  
- corrupted state was carried forward  
- truth fractured silently  

A correct major‑step produces **zero** irreproducible errors.

---

## 8. Major‑Steps Must Be Fully Reversible  
A major‑step must include:

- a forward path  
- a backward path  
- a stable checkpoint  
- a rollback plan  
- a rollback invariant  

If a major‑step cannot be reversed, it is not a major‑step.  
It is a leap — and leaps spawn monsters.

Correctness‑first developers do not leap.

---

## 9. Major‑Steps Must Be Observable  
A major‑step must be:

- logged  
- measured  
- monitored  
- verified  
- auditable  
- reproducible  

If the system cannot observe the major‑step, the system cannot guarantee correctness.

Unobserved transitions are monster factories.

---

## 10. Major‑Steps Must Be Boring  
A correct major‑step is:

- predictable  
- mechanical  
- uneventful  
- dull  
- safe  

If a major‑step feels exciting, dramatic, or heroic, it is wrong.

Correctness‑first engineering is boring.  
Boring is safe.  
Safe is correct.

---

## 11. Summary  
Major‑steps enforce correctness by:

- transitioning only between stable states  
- preserving all invariants  
- eliminating new sources of truth  
- avoiding snake pits  
- preventing irreproducible errors  
- ensuring reversibility  
- ensuring observability  
- building from micro‑steps  

Correctness‑first developers move systems forward in major‑steps.  
This is how professionals build systems that matter.
