## How to Fix Compiler Errors *Without Reading Them*  
### This Page Is About Getting Code to **Compile**  
### Not About Getting Code to **Work**

Before a program can *work*, it must *compile*.  
These are two completely different phases:

- **Phase 1: Get the code to compile.**  
  No syntax errors. No missing commas. No broken lines.

- **Phase 2: Get the code to work.**  
  Logic, correctness, behavior, output.

This page covers **Phase 1 only**.  
You cannot debug logic until the compiler is happy.

---

# What Is an Algorithm?
An **algorithm** is a *recipe*.  
You **follow it exactly**, step‑by‑step, without skipping, without guessing, without improvising.

Just like a cooking recipe:

- Step 1  
- then Step 2  
- then Step 3  
- in order  
- every time  

If you follow the recipe, you get the correct result.  
If you skip steps or mix them up, you get a mess.

The Beginner’s Algorithm is a **recipe for getting code to compile without reading compiler error messages**.

---

# Method 1 — Build From Scratch (Recommended for Beginners)  
This method is for students who:

- paste random code  
- write 5 goofy lines at once  
- cannot read compiler errors  
- cannot debug  
- cannot isolate failures  

This method **always works**.

---

## 1. Start with a clean F5 compile  
You must begin from a known‑good state.

If the program does **not** compile:

- remove the **last line(s) you added**  
- paste those lines into a text editor so you don’t lose them  
- press F5 again  
- if your program still won’t compile, start over, since you are back where you started and it still won't compile.

Only when F5 is clean can you begin the algorithm.

---

## 2. Write ONE line of code  
Not two.  
Not five.  
Not “finish the thought.”

**One line.**

If it requires multiple lines, such as if statement, write the minimum number of lines.
Beginners cannot debug multiple changes.

---

## 3. Press F5  
If it compiles → the line is correct.  
If it fails → the line you just wrote is a culprit.

You don’t need to read the error.  
You don’t need to interpret anything.  
You don’t need to guess.

You already know where to look.

---

## 4. Undo the line  
Beginners cannot fix broken lines yet.

So the rule is:

> **If F5 fails, undo the line you just wrote.**

Return to the last known good state.

If you want to keep the line for later, paste it into a text editor.

---

## 5. Try the smallest version of the idea  
Examples:

- Instead of writing a whole function call → write the variable declaration.  
- Instead of writing a loop → write it so it increments, stops, but has no body.  
- Instead of writing a struct → write one with no fields. (Yes, that is legal.)  
- Instead of writing logic → write a placeholder.

Press F5 again.

---

## 6. Now add the next smallest piece  
- If F5 fails, there’s a culprit in the last piece you added.  
- If F5 succeeds, you are closer to getting the whole line to compile.

---

## 7. Repeat until the full idea compiles  
You build the idea **one tiny step at a time**.  
Each step compiles.  
Each step is safe.  
Each step is guaranteed.

---

# Example — Build From Scratch (Finding the `'Age:'` Failure)

Beginner writes:

```d
writeln ("Name:", name, 'Age:', age, "Score:", score);
```

This fails to compile because `'Age:'` is an invalid character literal.
It should have double quotes as a string, not single quotes.

So we rebuild from scratch:

1. `writeln ();` → F5  
2. `writeln (name);` → F5  
3. `writeln ("Name:", name);` → F5  
4. `writeln ("Name:", name, 'Age:', age);` → F5  ← **this fails**  

Now we know the culprit is in the last piece we added:

```d
'Age:', age
```

We fix it:

```d
writeln ("Name:", name, "Age:", age);
```

Press F5 → it compiles.

5. Add `"Score:", score` last:

```d
writeln ("Name:", name, "Age:", age, "Score:", score);
```

Press F5 → it compiles.

This method is foolproof: the failure is discovered exactly at the step where `'Age:'` is introduced.

# Method 2 — Binary Chop (Recommended for Slightly More Advanced Students)  
Binary chop is typically used when the student wrote **multiple lines**, and the failure is somewhere in the mess.

Binary chop cuts the hiding surface **in half** every time.

---

## 1. Comment out half the code  
Use:

```d
/* ... */
```

Never inside a string or character literal.

Also check for comma mistakes:

- missing commas  
- extra commas  
- dangling commas at the end of argument lists  

These are extremely common beginner errors.

---

## 2. Press F5  
If it compiles → a bug is in the commented‑out half.  
If it fails → a bug is in the remaining half.

Guaranteed.

---

## 3. Chop again  
Take the failing half.  
Comment out half of *that*.  
Press F5.

---

## 4. Repeat until the exact line is isolated  
Binary chop finds the culprit quickly, even if the student cannot read the error message.

---

## 5. **Once the failing piece compiles, UNCOMMENT the hidden code**  
This is the part beginners often forget.

Binary chop temporarily hides code so you can isolate the failure.  
But once you fix the culprit (like `'Age:'`), you must:

1. **Uncomment the hidden code**  
2. Press F5  
3. Confirm the entire line now compiles

Binary chop is a *temporary hiding technique*, not a permanent deletion technique.

---

# Example — Binary Chop (Finding the `'Age:'` Failure)

Student writes:

```d
writeln ("Name:", name, 'Age:', age, "Score:", score);
```

It fails.

### Step 1 — Comment out half

```d
writeln ("Name:", name /*, 'Age:', age, "Score:", score */);
```

Press F5.

Assume this **compiles** → the bug is in the commented‑out half:

```d
'Age:', age, "Score:", score
```

### Step 2 — Chop inside the failing half

Replace the original line with only the failing half:

```d
writeln ('Age:', age, "Score:", score);
```

Now comment out half of this failing half:

```d
writeln ('Age:', age /*, "Score:", score */);
```

Press F5.

Assume this **fails** → the bug is in:

```d
'Age:', age
```

### Step 3 — Fix the culprit

```d
writeln ("Name:", name, "Age:", age, "Score:", score);
```

Press F5 → it compiles.

### Step 4 — **Uncomment the hidden code**

If any part of the original line is still commented out, restore it now.

Binary chop is complete.

# Summary  
This page is **only** about getting the code to *compile*.  
Getting the code to *work* is a **separate skill** taught later.

> **Build From Scratch**  
> - Recommended for beginners  
> - Start with a clean compile  
> - Write ONE minimal line  
> - Press F5  
> - If it fails, undo  
> - Add the next smallest piece  
> - Guaranteed to find the culprit (like `'Age:'`)

> **Binary Chop**  
> - Recommended for slightly more advanced students  
> - Comment out half  
> - Press F5  
> - If it compiles → bug in commented half  
> - If it fails → bug in remaining half  
> - Repeat until isolated  
> - **Once the failing piece compiles, UNCOMMENT the hidden code**  
> - Cuts hiding surface in half every iteration

You cannot debug logic until the compiler is happy.  
Compilation comes first.  
Correctness comes next.
