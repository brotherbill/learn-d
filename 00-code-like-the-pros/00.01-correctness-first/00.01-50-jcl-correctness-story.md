# 00.01.20 — JCL: Industrial‑Grade Correctness  
How Mainframe Discipline Prevents Monsters

Long before modern languages invented whiz‑bang features, mainframe shops were already practicing correctness‑first engineering.  
Their tool was **JCL — Job Control Language**.

JCL is not fancy.  
JCL is not modern.  
JCL is not “developer‑friendly.”

JCL is **correct**.

And correctness is the point.

---

## 1. JCL Requires Deterministic Thinking  
JCL forces the developer to think like a correctness‑first engineer:

- every step is explicit  
- every dependency is explicit  
- every input is explicit  
- every output is explicit  
- every failure mode is explicit  

There is no magic.  
There is no hidden state.  
There is no “eventual consistency.”  
There is no “retry later.”

JCL is deterministic.

---

## 2. JCL Requires a Failure Document  
In mainframe shops, correctness is not optional.  
If a job fails at step **N**, the developer must produce a document that says:

> **If the job fails at step N, here are the exact steps to take.**

Not vague steps.  
Not “try this.”  
Not “restart and hope.”  
Not “clear the cache.”  
Not “rerun the pipeline.”

Exact steps.

This document must be:

- correct  
- current  
- complete  
- reviewed
- deterministic  
- executable by an operator at 3 AM  

This is correctness‑first discipline.

---

## 3. The 3 AM Call  
I got the 3 AM call.

The job went down.  
Step N failed.

The operator asked the only question that matters:

> “Is the document current and accurate?”

The answer was:

- **Yes** — it is current.  
- **Yes** — it is accurate.

They followed the document.

Step by step.  
Exactly as written.  
No improvisation.  
No guessing.  
No debugging.  
No monster‑hunting.

And everything came back **A‑OK**.

The system recovered because correctness had already been done **during working hours**, not at 3 AM.

---

## 4. Correctness Happens Before Failure, Not After  
The next day, I went in and asked:

> “Did you follow the document?”

They said:

> “Yes. And it worked exactly.”

This is correctness‑first engineering:

- correctness is built **before** failure  
- correctness is documented **before** failure  
- correctness is verified **before** failure  
- correctness is rehearsed **before** failure  

When failure happens, correctness is already waiting.

There is no panic.  
There is no improvisation.  
There is no monster‑spawn.

---

## 5. Why JCL Is Correctness‑First  
JCL embodies correctness‑first principles:

- **Truth is singular.**  
  One job. One flow. One ledger of steps.

- **Invariants are explicit.**  
  Every step has preconditions and postconditions.

- **Failure is deterministic.**  
  If step N fails, the recovery path is known.

- **Operators do not guess.**  
  They execute the documented truth.

- **Developers do not hide behavior.**  
  There is no magic. No snake pits.

- **Systems do not continue after the impossible.**  
  A failed step halts the job — no undefined state.

JCL is not elegant.  
JCL is not fashionable.  
JCL is not modern.

JCL is **correct**.

---

## 6. The Lesson for Correctness‑First Developers  
Correctness‑first developers must adopt the JCL mindset:

- write down the exact recovery steps  
- ensure the document is current  
- ensure the document is accurate  
- ensure the document is executable  
- ensure the system halts on impossible conditions  
- ensure operators never guess  
- ensure truth never fractures  

Correctness is not what you do at 3 AM.  
Correctness is what prevents the 3 AM disaster.

---

## 7. Summary

> **JCL is correctness‑first engineering in its purest form:  
Explicit steps, explicit truth, explicit recovery, zero guessing.  
If the job fails at step N, the recovery path is already written.**

This is how professionals build systems that do not spawn monsters — even at 3 AM.
