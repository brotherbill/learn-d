# 00.01.20 — A Brief History of Correctness‑First  
How Software Lost Its Discipline — And Why Learn‑D Restores It

Correctness‑first is not new.  
It is the original discipline of software engineering — the one that built the most reliable systems in history.  
This file explains where correctness‑first came from, why it disappeared, and why it must return.

---

## 1. The 1960s: The Birth of Correctness‑First  
### Software for Space, Safety, and Survival

In the 1960s, software was used in places where failure meant death:

- spacecraft guidance  
- navigation systems  
- avionics  
- medical equipment  
- early nuclear control systems  

There were no GUIs, no frameworks, no “move fast and break things.”  
There was only discipline.

Every function had:

- clear **preconditions**  
- clear **postconditions**  
- clear **invariants**  
- clear **contracts**  
- incremental construction  
- stable foundations  

The Apollo Guidance Computer (AGC) is the canonical example:

> **No Apollo mission ever failed due to software error.**

Correctness‑first was not optional.  
It was survival.

---

## 2. The 1970s: Formal Methods and Discipline  
### The Rise of Structured Programming

The 1970s expanded correctness‑first with:

- structured programming  
- formal verification  
- Hoare logic  
- Dijkstra’s discipline  
- state machine reasoning  
- invariants as first‑class tools  

Software engineering was treated as mathematics:

- define the problem  
- define the contracts  
- define the invariants  
- prove correctness  
- then implement  

Correctness was the foundation of the field.

---

## 3. The 1980s: The Collapse  
### Cheap Hardware, Fast Compilers, and Management Pressure

In the 1980s, everything changed:

- hardware became cheap  
- compilers became fast  
- debugging became easy  
- deadlines became aggressive  
- management demanded speed  
- universities stopped teaching correctness  
- languages stopped enforcing contracts  

The new philosophy:

- “just try it”  
- “just push it”  
- “we’ll fix it later”  
- “debug in production”  

Correctness‑first was abandoned.  
Chaos replaced it.

This is when exponential error growth began to dominate the industry.

---

## 4. The 1990s–2000s: The Era of Fragile Systems  
### Features First, Correctness Last

As software expanded into:

- finance  
- medicine  
- transportation  
- communication  
- global commerce  

…the industry continued building on unstable foundations.

Symptoms:

- regressions everywhere  
- brittle systems  
- emergency patches  
- reconciliation departments  
- outages costing millions  
- compliance failures  
- customer distrust  

Correctness was treated as “too slow,” “too academic,” or “too expensive.”

The cost of this mistake is still being paid today.

---

## 5. The 2010s–2020s: Exponential Complexity  
### Systems Too Large to Debug

Modern systems are:

- distributed  
- concurrent  
- stateful  
- interconnected  
- mission‑critical  
- globally deployed  

Incorrect systems now fail in ways that are:

- unpredictable  
- cascading  
- exponential  
- financially catastrophic  
- safety‑critical  

Debugging does not scale.  
Human cognition does not scale.  
Exponential error growth overwhelms linear developer capacity.

The industry is finally realizing:

> **Correctness‑first is not optional.  
It is the only sustainable path forward.**

---

## 6. The Present: The Return of Correctness‑First  
### Eiffel, Formal Methods, and Mission‑Critical Engineering

A few disciplines preserved correctness‑first:

- Eiffel’s Design by Contract  
- formal verification  
- aerospace engineering  
- medical device software  
- nuclear control systems  
- high‑assurance systems  

These fields never abandoned correctness because they could not afford to.

Now the rest of the industry is being forced to return to correctness‑first because:

- outages cost millions  
- errors multiply exponentially  
- reconciliation departments grow  
- regulators intervene  
- customers lose trust  
- companies face bankruptcy  

Correctness‑first is returning because the economics demand it.

---

## 7. The Future: Correctness‑First Developers Are Essential  
### The World You Will Inherit

The next generation of developers will be asked to:

- repair broken systems  
- stabilize brittle foundations  
- eliminate cascading errors  
- restore invariants  
- rebuild contracts  
- make existing systems correct  

Management will pay the costs because the alternative is bankruptcy.

Correctness‑first developers will be:

- rare  
- respected  
- essential  
- highly paid  
- mission‑critical  

The world needs correctness‑first engineers because the world now runs on software that cannot afford to fail.
