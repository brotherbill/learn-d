# 00.01.05 — Preconditions, Postconditions, and Invariants  
Correctness‑First Foundations

Preconditions, postconditions and invariants are the simplest and most powerful tools for ensuring correctness in software. They were standard practice in the 1960s, abandoned in the 1980s, and are now almost unknown to modern developers. Learn‑D restores them because correctness cannot exist without them.

This document defines these terms, explains why they matter, and shows how they fit into the correctness‑first discipline.

---

## 1. Definitions (all terms defined before use)

### Preconditions  
A **precondition** is something that must be true *before* a function runs.  
If a precondition is not met, the function cannot guarantee correct behavior.

Examples (conceptual, not code):
- “The input must be a positive number.”
- “The file must exist.”
- “The buffer must have space.”

### Postconditions  
A **postcondition** is something that must be true *after* a function finishes.  
If a postcondition is not met, the function has failed, even if it ran without crashing.

Examples:
- “The result must be sorted.”
- “The output must match the specification.”
- “The state must be updated exactly once.”

### Invariants  
An **invariant** is something that must remain true *throughout* key points during the execution of a function, loop, or system.  
If an invariant is ever violated, the system is in an incorrect state — even if no error message appears.

Examples:
- “The index must always remain within bounds.”
- “The balance must never become negative.”
- “The state machine must always be in one of the defined states.”

Invariants define what must **never** change.  
They are the backbone of correctness.

### Contract  
A **contract** is the combination of preconditions, postconditions, and invariants.  
It defines what a function promises, what the caller must guarantee, and what the system must maintain.

---

## 2. Why Preconditions, Postconditions, and Invariants Matter

These three tools force the programmer to think about:

- what must be true before the function runs  
- what must be true after the function runs  
- what must remain true during execution  
- what the function guarantees  
- what the caller must guarantee  
- what the system must guarantee  

This eliminates ambiguity.  
This eliminates guesswork.  
This eliminates “maybe it works.”  
This eliminates “it seems fine.”  
This eliminates “we’ll fix it later.”

Correctness is not magic.  
Correctness is not luck.  
Correctness is not “hope the compiler catches it.”

Correctness is built by **contracts**.

---

## 3. Why the 1960s Used Preconditions, Postconditions, and Invariants

In the 1960s:

- software controlled spacecraft  
- software controlled navigation  
- software controlled life‑support  
- software controlled guidance computers  

There was no room for “maybe.”  
There was no room for “probably.”  
There was no room for “we’ll patch it later.”

Every function had:

- clear preconditions  
- clear postconditions  
- clear invariants  
- clear contracts  

This discipline produced the Apollo Guidance Computer — a system that never failed due to software error.

Correctness‑first was not optional.  
It was survival.

---

## 4. Why the 1980s Abandoned Them

In the 1980s:

- hardware became cheap  
- compilers became fast  
- debugging became easy  
- deadlines became aggressive  
- management demanded speed  
- universities stopped teaching correctness  
- languages stopped enforcing contracts  

The industry replaced correctness with:

- “just try it”  
- “just push it”  
- “we’ll fix it later”  
- “debug in production”  

Contracts were seen as “too slow,” “too formal,” or “too academic.”

The result was predictable:

- unstable systems  
- brittle codebases  
- endless regressions  
- reconciliation departments  
- permanent firefighting  

Correctness was abandoned.  
Chaos replaced it.

---

## 5. Dr. Bertrand Meyer’s Principle: Correctness Before Performance

Dr. Bertrand Meyer, creator of Eiffel and Design by Contract, expressed a core truth of software engineering:

> **If a system is not correct, its performance is irrelevant.**

A fast system that produces the wrong answer is not “high‑performance” — it is **wrong at high speed**.  
An optimized system that violates its invariants is not “efficient” — it is **efficiently incorrect**.

Correctness is the foundation.  
Performance is a refinement.

This aligns directly with correctness‑first:

- correctness defines what the system must do  
- performance defines how fast it does it  
- correctness violations invalidate all performance claims  

First: make the system correct.  
Then: make the correct system fast.

---

## 6. What Happens When Contracts Are Practiced

When preconditions, postconditions, and invariants are used:

- functions become predictable  
- bugs become rare  
- debugging becomes mechanical  
- systems become stable  
- regressions disappear  
- correctness becomes normal  

This is how the moon landing happened.  
This is how avionics work.  
This is how medical systems stay safe.  
This is how financial systems avoid disaster.

Correctness‑first produces reliability.

---

## 7. What Happens When They Are Not Practiced

When contracts are ignored:

- functions behave unpredictably  
- bugs multiply  
- regressions cascade  
- systems break  
- outages become normal  
- teams panic  
- reconciliation departments grow  

Reconciliation departments exist because correctness was not practiced.

They are the cost of abandoning contracts.
