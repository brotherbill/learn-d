# 00.01.03 — Spawning Grounds of Little Monsters  
Where Incorrect Systems Create Multiple Sources of Truth

Correctness‑first developers must learn to recognize every place where systems can spawn **little monsters** — the mismatched records, inconsistent states, contradictory ledgers, and fractured truths that destroy correctness.

These monsters do not appear randomly.  
They are born in predictable hiding places inside incorrect systems.

This chapter teaches you where they come from.

---

## 1. Monsters Are Born When Truth Forks  
### Any time truth is duplicated, cached, or recomputed independently, monsters appear.

A system is correct only when **truth is singular**.

A system becomes incorrect the moment truth is:

- copied  
- cached  
- recomputed  
- approximated  
- delayed  
- “eventually consistent”  
- stored in multiple places  
- updated by multiple processes  

Every one of these actions creates a **spawning ground** for little monsters.

Correctness‑first developers learn to identify these places immediately.

---

## 2. Misused Systems Create Monsters  
### Many tools are safe when used correctly — and catastrophic when misused.

A correctness‑first developer must understand not only how systems work, but how they are commonly misused.

Misuse is the birthplace of monsters.

Examples:

- caching values that must remain authoritative  
- storing truth in multiple databases  
- using asynchronous updates for critical state  
- recomputing values instead of preserving invariants  
- allowing race conditions to determine truth  
- using “eventual consistency” for customer‑visible data  

These misuses create contradictions that reconciliation teams must later repair.

Correctness‑first developers prevent the contradictions from ever being born.

---

## 3. The Caching Disaster  
### Caching is powerful — and dangerous.

Caching is an excellent technique when used correctly:

- downloading images once  
- storing static assets  
- avoiding repeated expensive computations  

But caching is catastrophic when used incorrectly.

Your **bank account balance** should never be cached.  
Your **financial ledger** should never be “eventually consistent.”  
Your **customer‑visible truth** should never fork.

Customers are not amused when:

- Mom’s smartphone says the balance is **$200**  
- Dad’s desktop says the balance is **$5,200**

This is not a “performance optimization.”  
This is a monster.

Caching creates monsters when:

- the cached value is authoritative  
- the cached value can change  
- the cached value is customer‑visible  
- the cached value is financial  
- the cached value is regulatory  
- the cached value is part of an invariant  

Correctness‑first developers know exactly when caching is safe — and when it is a snake pit.

---

## 4. Eventual Consistency Is a Monster Factory  
### “Eventually consistent” means “eventually wrong.”

Eventual consistency is often marketed as a modern, scalable technique.

In reality, it is a **monster factory**.

Eventual consistency means:

- truth is allowed to fork  
- truth is allowed to drift  
- truth is allowed to contradict itself  
- truth is allowed to be wrong for a while  

Correctness‑first developers reject eventual consistency for:

- financial systems  
- customer‑visible systems  
- regulatory systems  
- safety‑critical systems  
- any system where correctness matters  

Truth must be **always consistent**, not “eventually.”

---

## 5. Race Conditions Spawn Monsters  
### When two processes compete to write truth, truth fractures.

Race conditions create monsters because:

- two writers produce two truths  
- the winner is random  
- the loser is forgotten  
- invariants are violated  
- downstream systems inherit contradictions  

Correctness‑first developers eliminate races by:

- enforcing strict ordering  
- using correct locking  
- designing deterministic workflows  
- preventing concurrent writes to authoritative truth  

Monsters cannot survive in deterministic systems.

---

## 6. Multiple Databases = Multiple Truths  
### Every database is a universe. Two universes cannot share truth.

When truth is stored in:

- two databases  
- two services  
- two microservices  
- two caches  
- two replicas  
- two ledgers  

…it will eventually diverge.

Even if the divergence is small, it grows.  
Even if the divergence is rare, it spreads.  
Even if the divergence is unnoticed, it becomes catastrophic.

Correctness‑first developers insist on:

- one authoritative database  
- one authoritative ledger  
- one authoritative source of truth  

Everything else is a copy — and copies are monsters waiting to hatch.

---

## 7. Monsters Hide in Undefined Behavior  
### Undefined behavior is a breeding ground for contradictions.

Undefined behavior means:

- the system can do anything  
- the system can contradict itself  
- the system can corrupt state  
- the system can violate invariants  

Undefined behavior is not a “corner case.”  
It is a spawning ground.

Correctness‑first developers eliminate undefined behavior entirely.

---

## 8. Try/Catch as a Breeding Ground  
### If NASA won’t use it, there’s a reason.

Try/catch looks safe.  
It looks modern.  
It looks convenient.

But try/catch is a **monster incubator** when used incorrectly.

Why?

Because try/catch:

- hides the source of failure  
- hides the location of failure  
- hides the cause of failure  
- allows execution to continue in an undefined state  
- allows invariants to be violated silently  
- allows corrupted data to propagate  
- allows truth to fracture without detection  

This is why **NASA does not use exceptions** in flight‑critical software.

If NASA won’t use it, correctness‑first developers should ask why.

The answer is simple:

> **Exceptions allow systems to continue running after something impossible has happened.  
That is the perfect spawning ground for little monsters.**

Correctness‑first developers use explicit error handling:

- clear return codes  
- explicit failure paths  
- explicit state correction  
- explicit invariant restoration  
- explicit truth preservation  

Exceptions hide monsters.  
Explicit handling prevents them.

---

## 9. Most Errors Are Not Reproducible  
### And that is a warning sign, not a statistic.

In many large organizations, only **about 30% of error tickets are reproducible and resolved**.  
The remaining 70%:

- cannot be reproduced  
- cannot be diagnosed  
- cannot be traced  
- cannot be explained  
- cannot be fixed  

They simply “go away,” or worse — they return later in a different form.

This is not normal.  
This is not acceptable.  
This is not “just how software works.”

This is what happens when systems spawn little monsters.

Unreproducible errors mean:

- undefined behavior is present  
- invariants are being violated  
- truth is fracturing  
- corrupted state is being carried forward  
- error handling is hiding the real failure  
- the system is continuing after something impossible has happened  

And this ties directly to the danger of try/catch:

> **If an error cannot be reproduced, it means the system continued running in an undefined state.  
Undefined state is where little monsters are born.**

This is why **NASA refuses to use exceptions** in flight‑critical software.

If NASA won’t use it, correctness‑first developers should ask why.

The answer is simple:

> **Unreproducible errors are evidence of monsters hiding in the system.  
Correctness‑first developers eliminate the hiding places.**

---

## 10. The Mission of Correctness‑First Developers  
### Find every hiding place where monsters can be born — and seal it.

Correctness‑first developers must:

- identify every place truth can fork  
- eliminate every misuse of caching  
- reject eventual consistency for critical truth  
- remove race conditions  
- prevent concurrent writes  
- enforce invariants  
- maintain contracts  
- ensure truth is singular  
- ensure truth is stable  
- ensure truth is correct  

Your job is not to fix monsters after they appear.  
Your job is to prevent them from ever being born.

---

## 11. Summary

> **Little monsters — multiple sources of truth — are born when systems misuse tools, fork truth, hide errors, or allow inconsistency.  
Correctness‑first developers learn to identify every spawning ground and eliminate it.**

This is how professionals build systems that do not betray their users, their operators, or their data.
