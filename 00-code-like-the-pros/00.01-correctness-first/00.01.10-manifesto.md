# 00.01.10 — Correctness‑First Manifesto  
Mission Critical Foundations

Software is now everywhere — in medicine, aviation, finance, transportation, communication, and daily life. Yet the discipline that once made software trustworthy has been largely forgotten. This manifesto restores the principles that produced the most reliable systems in history.

Correctness‑first is not a slogan.  
Correctness‑first is a discipline.  
Correctness‑first is the foundation of professional software.

---

## 1. The First Principle: Correctness Before Everything

A program must be **correct** before it is anything else.

- Before it is fast  
- Before it is elegant  
- Before it is optimized  
- Before it is scalable  
- Before it is “feature‑rich”  
- Before it is shipped  

Correctness is the foundation.  
Everything else is decoration.

Dr. Bertrand Meyer expressed this with perfect clarity:

> **If a system is not correct, its performance is irrelevant.**

A wrong answer delivered quickly is still wrong.  
A broken system running efficiently is still broken.  
A fast failure is not a success.

Correctness is the prerequisite for performance, reliability, and trust.

---

## 2. The Stable Foundation

Correctness begins with a stable foundation:

> **A program that compiles, runs, and produces correct output.  
> All new code must be built on top of this.**

This principle prevents:

- cascading errors  
- ghost bugs  
- regressions  
- brittle systems  
- emergency patches  
- reconciliation departments  

A stable foundation is the anchor of correctness‑first development.

---

## 3. Contracts: Preconditions, Postconditions, and Invariants

Correctness is not accidental.  
Correctness is constructed.

Contracts are the tools that make correctness possible:

- **Preconditions** — what must be true before execution  
- **Postconditions** — what must be true after execution  
- **Invariants** — what must remain true throughout execution at key points

These tools force clarity, eliminate ambiguity, and prevent entire classes of errors.

Contracts were standard practice in the 1960s.  
They were abandoned in the 1980s.  
Learn‑D restores them.

---

## 4. The Historical Proof: The Apollo Guidance Computer

The Apollo Guidance Computer (AGC) is the strongest evidence for correctness‑first engineering.

- Every routine had defined preconditions  
- Every routine had defined postconditions  
- Every subsystem had invariants  
- Every interface had a contract  
- Every change was incremental  
- Every build was stable  

The result:

> **No Apollo mission ever failed due to software error.**

Correctness‑first put humans on the moon.

---

## 5. The Collapse: The 1980s and After

In the 1980s, the industry abandoned correctness:

- hardware became cheap  
- compilers became fast  
- debugging became easy  
- deadlines became aggressive  
- universities stopped teaching correctness  
- languages stopped enforcing contracts  

The new philosophy:

- “just try it”  
- “just push it”  
- “we’ll fix it later”  
- “debug in production”  

The consequences:

- unstable systems  
- brittle codebases  
- endless regressions  
- permanent firefighting  
- reconciliation departments everywhere  

Correctness was replaced by speed.  
Quality was replaced by hope.

---

## 6. The Cost of Incorrect Software

Incorrect software creates:

- outages  
- data loss  
- financial errors  
- safety failures  
- emergency patches  
- customer distrust  
- engineering burnout  

Incorrect software creates **reconciliation departments** — teams whose job is to fix the consequences of incorrect systems.

Correctness‑first eliminates the need for reconciliation departments.
